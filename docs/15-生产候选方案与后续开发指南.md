# 生产候选方案与后续开发指南

> 本文件记录 2026-06-23 量化审计后的最终工程口径。
> 它不是投资承诺，也不是实盘杠杆授权。它是后续修改 Python、API、前端和文档时必须遵守的项目宪法。

## 一、最终判定

当前结论：

- `Research PASS`
- `Production design APPROVED`
- `Live leveraged execution NOT YET APPROVED`

含义：

- 研究层面，干净 Yahoo Adj Close 数据、旁路回测、归因审计、组合聚合审计已经形成闭环。
- 生产设计层面，可以把 `90% 防御核心 + 10% 现代贝塔卫星` 作为影子观察候选方案。
- 实盘层面，仍然不允许直接上杠杆，不允许默认真实下单，不允许把 Portfolio Margin 理论安全边界当成券商真实保证。

下一阶段不是加杠杆，而是连续 60 个交易日做 `Shadow Sync & Structural Audit`。

## 三、关键审计结论

所有历史回测必须使用干净正价格数据。AkShare 美股 `qfq` 前复权曾在长周期分红标的上生成负价格，导致伪造收益。任何非正价格都必须 fail closed。

固定防御核心在干净数据与 50bp 恐慌滑点下：

- CAGR：约 `6.27%`
- MDD：约 `-11.29%`

90/10 聚合候选在干净数据下：

- CAGR：约 `7.60%`
- MDD：约 `-13.95%`
- 结论：研究通过，允许进入影子同步，不允许直接实盘杠杆。

不要再继续在以下方向做无约束参数扫描：

- 为了追求漂亮曲线继续调 MA 窗口
- 为了修复 CAGR 随意替换 TLT 为 IEF/AGG
- 为了压成本过度修改恢复期
- 为了捕捉 2022 把波动率/相关性阈值调得过低

这些方向已经审计过，边际收益不足，过拟合风险更高。

## 四、Stage 9.5 影子观察

当前阶段只允许观察，不允许真实交易。

### 1. 统一影子审计周期

命令：

```bash
python -m live.stage95_shadow_runner --aum 2000000 --no-broker --skip-db
```

接通 IBKR Client Portal 后：

```bash
python -m live.stage95_shadow_runner --aum 2000000 --broker ibkr
```

输出：

- `data/shadow/shadow_sync_YYYYMMDD_HHMMSS.json`
- `data/shadow/latest_shadow_sync.json`
- `data/shadow/margin_snapshot_YYYYMMDD_HHMMSS.json`
- `data/shadow/latest_margin_snapshot.json`
- `data/shadow/stage95_cycle_YYYYMMDD_HHMMSS.json`
- `data/shadow/latest_stage95_cycle.json`
- 可选写入 `shadow_run_reports`

用途：

- 计算 90/10 生产候选目标权重和目标名义本金
- 捕获券商报价或本地 Adj Close 兜底报价
- 估算 bid/ask 半价差滑点
- 有真实账户快照时生成 shadow orders
- 只读抓取券商保证金快照
- 形成完整 Stage 9.5 周期级状态
- 永远不提交订单

### 2. 单项调试入口

如需单独排查影子账本或保证金快照，可以直接运行底层脚本。

影子同步：

```bash
python -m live.shadow_sync --aum 2000000 --no-broker --skip-db
```

保证金监控：


```bash
python -m live.margin_monitor --broker ibkr --skip-db
```

输出：

- 提取 `FullMaintainMarginReq / ExcessLiquidity / NetLiquidation / BuyingPower / Leverage` 等字段
- 如果字段缺失，状态必须是 `UNAVAILABLE` 或 `PARTIAL`
- 不允许从缺失数据中推导安全结论
- `latest_*` 报告超过 24 小时未更新时，API 必须标记为 stale 并要求人工关注

### 3. 60 日观察汇总

命令：

```bash
python -m live.stage95_observation_summary --shadow-dir data/shadow --expected-cycles 60
```

输出：

- `data/shadow/stage95_observation_summary_YYYYMMDD_HHMMSS.json`
- `data/shadow/latest_stage95_observation_summary.json`

用途：

- 统计已观察周期数与 60 日覆盖率
- 统计 `ATTENTION / CRITICAL / STALE` 周期数
- 统计券商不可用次数与连续异常 streak
- 统计最大/平均半价差 bps
- 统计 `NetLiquidation / ExcessLiquidity / FullMaintainMarginReq` 保证金字段覆盖率
- 明确保留 `live_leverage_approved = false`

## 五、Python 开发准则

### 1. 分层边界

- `engine/` 放生产可复用逻辑。
- `backtest/` 放旁路实验和审计脚本。
- `live/` 放生产前只读观察脚本。
- `runner/` 放日常任务入口。
- `api/` 只负责把已存在的账本和审计记录暴露给前端。
- `frontend/` 只做只读呈现和治理申请，不做投资决策。

新策略先进入 `backtest/`，通过审计后再抽公共逻辑进 `engine/`。不要把旁路实验直接塞进生产引擎。

### 2. 数据口径

所有价格入口必须满足：

- 美股 ETF 优先 Yahoo Adj Close。
- 禁止使用会生成负价格的美股 AkShare `qfq` 结果。
- 任何非正价格、空价格、日期错位，必须 fail closed。
- 回测和 live 观察都要保留数据来源和 as-of 日期。
- `data/raw/*.csv` 是可再生成本地缓存，默认不进入普通 PR。
- 已批准的研究复现输入放入 `data/audit_snapshots/<date>-<source>/`，当前冻结快照为 `data/audit_snapshots/2026-06-23-yahoo-adj-close/`。
- 冻结快照目录必须自带 `manifest.json`，记录行数、日期范围和 SHA256；`data/data_manifest.json` 与 `data/data_status.json` 记录可再生成 raw 缓存状态。

### 3. 执行边界

默认禁止真实交易。

只有满足全部条件，才允许讨论小额真实验证：

- `XIRANG_ENABLE_LIVE_CORE_EXECUTION=1` 显式开启 Core 真实执行总闸门
- `XIRANG_LIVE_CORE_APPROVAL_ID` 写入人工批准引用
- `IBKR_ENABLE_ORDER_SUBMISSION=1` 或对应券商开关显式开启
- 资产在白名单内
- 订单通过执行前硬闸门
- 当日券商同步与对账覆盖交易日
- 执行后回执与本地账本闭环
- 出金仍然只能人工审批和人工执行

任何影子脚本不得调用：

- `place_order`
- `cancel_order`
- `get_order_status`，除非是在专门的订单审计脚本中读取已存在订单

### 4. 测试要求

新增功能至少覆盖：

- 正常路径
- 缺失数据
- 非正价格或异常价格
- 券商不可达
- fail-closed 状态
- 不调用真实交易方法

关键测试命令：

```bash
python -m pytest -q
python -m pytest tests/test_shadow_sync_live.py tests/test_margin_monitor_live.py -q
```

## 六、前端开发准则

当前前端是静态只读面板：

- `frontend/index.html`：页面结构
- `frontend/app.js`：状态获取、渲染、交互
- `frontend/i18n.js`：中英文文案
- `frontend/style.css`：自定义样式
- `frontend/chart.min.js` 与 `frontend/tailwind-local.css`：本地静态依赖

### 1. 前端定位

前端不是交易操作台。

允许：

- 展示 NAV、回撤、持仓、风控事件
- 展示 Broker Sync / Shadow Run / Margin Snapshot 健康状态
- 展示警告、缺失数据、连续异常天数
- 发起出金申请
- 展示人工治理和审计链

不允许：

- 一键下单
- 一键加杠杆
- 一键从 Shadow 切 Live
- 用绿色大按钮刺激交易行为
- 隐藏 `UNAVAILABLE / WARNING / FAIL_CLOSED`

### 2. 新增前端功能的正确顺序

新增一个观察模块时，按这个顺序做：

1. 后端先落结构化 JSON。
2. 数据库或文件输出先有测试。
3. FastAPI 路由只读暴露该 JSON。
4. `frontend/i18n.js` 增加中英文文案。
5. `frontend/app.js` 渲染只读状态。
6. `frontend/style.css` 补最小样式。
7. 用浏览器检查移动端和桌面端不溢出、不遮挡。

不要让前端直接读取本地文件，也不要让前端自己计算风控结论。前端只展示后端已经审计过的状态。

### 3. 视觉和交互原则

- 风控状态优先于收益数字。
- `UNAVAILABLE` 要比 `OK` 更显眼。
- 影子观察要明确写出“不产生真实交易”。
- 保证金监控要明确写出“观察不等于杠杆批准”。
- 图表可以减少，但警告不能隐藏。
- 移动端必须能看清状态和最后更新时间。

### 4. 已接入的 Stage 9.5 前端区域

dashboard 已经接入 `Stage 9.5 Shadow Audit` 只读区域，数据来源为：

- `/api/shadow-audit/{portfolio_id}`
- `/api/stage95-observation-summary/{portfolio_id}`

该区域展示：

- 最新 `shadow_sync` 状态
- 90/10 目标权重
- 目标名义本金
- 报价来源
- 最大观测半价差 bps
- 最新 `margin_snapshot` 状态
- `ExcessLiquidity / NetLiquidation / FullMaintainMarginReq`，如果券商提供
- 连续 `UNAVAILABLE / WARNING` 天数

该区域必须保持只读。缺失、过期、不可解析、券商不可达或保证金字段缺失时，必须展示 `MISSING / STALE / WARNING / UNAVAILABLE / ATTENTION` 一类需要人工关注的状态，不能渲染成安全状态。不得增加一键交易、一键加杠杆或 Shadow 转 Live 控件。

## 七、禁止事项

这些事情不要做：

- 不要把研究 PASS 写成实盘杠杆 PASS。
- 不要把 `latest_margin_snapshot.status = UNAVAILABLE` 展示成“暂无风险”。
- 不要为了提高 CAGR 把 10% 卫星仓位偷偷调成 20%。
- 不要把 `SMH` 加回生产候选，除非重新审计。
- 不要把 Shadow/Sandbox 收益混入正式收益。
- 不要把前端做成刺激交易的 cockpit。
- 不要在没有真实券商快照时估算“不会爆仓”。

## 八、下一步路线

建议顺序：

1. 连续 60 个交易日运行 `live.stage95_shadow_runner`。
2. 每日运行 `live.stage95_observation_summary` 汇总覆盖率、滑点、保证金字段和不可用次数。
3. 固化 Stage 9.5 影子观察运行手册，写清每日运行、异常判定、人工介入和 60 日验收标准。
4. 接通真实券商只读环境后，先验证保证金字段覆盖率、报价来源、半价差和连续不可用 streak。
5. 完成 60 日观察审计后，再决定是否进入小额、无杠杆或极低杠杆真实验证。

这套系统的核心价值不是更激进地赚钱，而是让大资金在不确定的真实世界里少犯一次不可逆的大错。
