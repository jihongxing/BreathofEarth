# MVP 验证计划

## 目标

用最少的代码验证一个核心问题：

> 这个系统在不同市场环境下，会不会做出愚蠢决策？

不是验证"能不能赚钱"，而是验证"行为是否稳定"。

---

## Phase 0：数据准备（1-2天）

### 需要的数据

| 资产 | 代表标的 | 用途 | 数据源 |
|------|---------|------|--------|
| 美股大盘 | SPY | Core 层股票 | Yahoo Finance |
| 长期国债 | TLT | Core 层债券 | Yahoo Finance |
| 黄金 | GLD | Core 层避险 | Yahoo Finance |
| 短期国债/现金 | SHV / BIL | Stability 层 | Yahoo Finance |
| 通胀数据 | CPI | 验证抗通胀能力 | FRED |

### 数据要求

- 日频收盘价（adjusted close）
- 时间跨度：2005-2025（覆盖08金融危机、20年疫情、22年股债双杀）
- 格式：CSV 或直接用 yfinance 拉取

### 交付物

- `data/` 目录下的 CSV 文件
- 一个 `data_loader.py` 负责加载和清洗

## Phase 1：最简回测（3-5天）

### 目标

实现永久组合的最简回测，验证基础策略的历史表现。

### 实现内容

1. 简单的等权组合（25% SPY + 25% TLT + 25% GLD + 25% SHV）
2. 按阈值再平衡（偏离 > 20% 触发）
3. 按时间再平衡（每年强制一次）
4. 计算核心指标

### 核心指标

| 指标 | 说明 | 成功标准 |
|------|------|---------|
| 年化收益率（CAGR） | 长期复合增长 | ≥ 通胀 + 2% |
| 最大回撤（MDD） | 最坏情况亏损 | ≤ -15% |
| 波动率 | 年化标准差 | < 股票的一半 |
| 夏普比率 | 风险调整收益 | > 0.5 |
| 超通胀年份占比 | 购买力保护 | > 70% |

### 交付物

- `backtest/simple_backtest.py`
- 输出：收益曲线图、回撤曲线图、指标汇总表

---

## Phase 2：状态机回测（5-7天）

### 目标

在 Phase 1 基础上加入状态机和风控逻辑，验证系统行为。

### 新增内容

1. Portfolio 状态机（IDLE / REBALANCING / PROTECTION）
2. 三层资金分配（Core 75% / Stability 15% / Alpha 10%）
3. 风控规则：
   - 回撤 > -15% → 进入 PROTECTION
   - 相关性崩溃检测（股债30日滚动相关性 > 0.5）
4. Alpha 层简单模拟（可用动量策略或直接跳过）
5. 每步记录状态快照

### 关键验证点

- 2008年：系统是否触发保护？Core 层回撤多少？
- 2020年3月：系统反应速度如何？
- 2022年：股债双杀时系统行为是否合理？
- 风控触发次数是否合理（不过多也不过少）？
- 调仓频率是否在预期范围内？

### 交付物

- `engine/portfolio.py` — 状态机
- `engine/risk.py` — 风控引擎
- `engine/rebalance.py` — 再平衡逻辑
- `backtest/stateful_backtest.py` — 带状态机的回测
- 输出：状态转移时间线、风控触发记录、各层权重变化图

---

## Phase 3：仿真运行（Phase 2 通过后）

### 目标

接入实时行情，Paper Trading 模式运行，验证系统在真实市场节奏下的稳定性。

### 实现内容

1. 用 yfinance 或免费 API 拉取每日行情
2. 状态机每日运行一次
3. 生成调仓指令但不执行（记录到数据库）
4. 每日记录 NAV、权重、偏离度

### 运行周期

至少 1-3 个月，观察系统行为。

### 交付物

- `runner/daily_runner.py` — 每日执行脚本
- SQLite 数据库记录所有状态
- 简单看板（Streamlit 或 HTML）

## 决策节点

### Phase 1 完成后判断

- 如果永久组合历史表现不满足成功标准 → 调整资产配置比例或标的
- 如果满足 → 进入 Phase 2

### Phase 2 完成后判断

- 如果状态机在极端环境下行为异常 → 调整风控参数
- 如果风控触发过于频繁 → 放宽阈值
- 如果一切合理 → 进入 Phase 3

### Phase 3 运行1-3个月后判断

- 如果系统稳定 → 考虑接入券商 API，小资金实盘
- 如果发现问题 → 回到 Phase 2 修正

---

## 防坑清单

1. 不要在回测阶段调参数去"拟合"历史 → 用固定规则
2. 回测必须包含滑点（0.05%-0.2%）和手续费
3. 不要只看收益，重点看回撤和行为稳定性
4. 不要跳过 Phase 3 直接实盘
5. 不要过早加复杂功能（多成员、权限、前端美化）
