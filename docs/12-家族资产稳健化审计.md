# 家族资产稳健化审计

> 目标不是更高收益，而是更少犯错。
> 息壤的核心任务，是用纪律、时间和数学规律，让家族资产长期跑赢通胀，而不是追逐高增长。

## 一、审计结论

这套系统的方向是对的，但按"家族资产长期运行"标准看，当前仓库仍处于 `可验证原型` 阶段，还不是 `可放心托付` 阶段。

核心问题不在于策略不够激进，而在于边界还不够硬：

- 管理员初始化仍有被抢占风险
- 出金治理仍有被拆单绕过的空间
- 数据新鲜度还没有被当作硬门槛
- 文档承诺与当前可执行结果存在偏差
- 实验层与稳定层仍未做到彻底剥离

结论很明确：

- 入金可以自动化
- 出金不能自动化
- Alpha/Lab 必须从家族主资产体系中剥离

## 二、三条硬原则

### 1. 稳健增长永远大于高增长

- 系统优先级固定为：`稳定 > 抗通胀 > 收益`
- 不允许为了多赚 1-2 个百分点，把系统暴露在结构性失误风险里
- 任何新增功能，都要先回答"是否会增加误操作概率"

### 2. 出金只能请求，不能自动执行

- 系统可以提供 `withdraw preview`
- 系统可以记录 `withdraw request`
- 系统可以生成审批和审计记录
- 系统不应自动批准任何出金
- 系统不应自动执行任何出金
- 系统不应直接把出金请求变成券商卖出或资金划转

家族资产体系里，出金是治理行为，不是交易行为。

因此正确流程应为：

```text
发起出金请求 -> 冷静期 -> 多人审批 -> 人工执行 -> 人工回填结果 -> 审计留痕
```

### 3. 实验层必须与稳定层彻底剥离

- `Core + Stability` 才是家族主资产
- `Alpha / Lab` 只能是独立实验仓
- 实验仓不能与主仓共用同一套净值口径
- 实验仓不能直接写入主仓风控结果
- 实验仓不能通过共享状态、共享表、共享默认账户污染主仓
- 实验仓即使盈利，也不能默认自动回流主仓，必须经过人工复核和治理批准

一句话：`Lab 可以死，Core 不能被 Lab 拖下水。`

## 三、推荐的目标架构

审计后，系统应明确拆成四个区域：

### A. Core

家族资产主引擎，只负责：

- 市场数据读取
- 数据校验
- 风控判断
- 状态机调仓
- Core/Stability 资金流
- 审计记录

禁止放入：

- Alpha 策略
- 竞技场
- 网格、动量、备兑等实验逻辑
- 任何会改变主仓风险偏好的探索功能

### B. Governance

治理层只负责：

- 入金确认
- 出金请求
- 多签审批
- 冷静期
- 审计日志

强制原则：

- 入金可自动入账
- 出金只能人工处理
- 任何金额的出金都不能直接自动批准
- 必须防拆单，按日/周/月累计口径限制提款

### C. Lab

实验层单独存在，建议独立：

- 数据表
- API 前缀
- 进程
- 账户
- 净值曲线
- 审计口径

主仓只能"看见"实验层的汇总报告，不能被实验层直接改写。

### D. Sidecar

附属优化层，例如：

- 税损收割
- TWAP
- 报告生成

这些模块只能在满足两个条件时存在：

- 它们失败不会伤害 Core
- 它们不会改变主仓的风险边界

## 四、本轮审计的主要发现

### P0. 首次部署存在控制权被抢占风险

问题：

- 空库时可通过未认证接口初始化第一个管理员

风险：

- 互联网暴露环境下，任何首个调用者都可能永久接管系统

要求：

- 首次管理员初始化必须改为本地 CLI、一次性 secret，或仅限 localhost

### P1. 当前出金治理不符合"克制取款"原则

问题：

- 低于阈值的小额出金可自动批准
- 没有按日/周/月累计限额
- 没有反拆单规则

风险：

- 可通过连续小额请求绕过家族治理机制

要求：

- 所有出金至少都要经过请求登记
- 大额出金必须多人审批 + 冷静期
- 小额出金至少要纳入滚动累计额度和人工确认
- 默认不自动执行

### P1. 数据新鲜度还不是硬门槛

问题：

- 系统记录了 `last_update.txt`
- 但没有强制要求最新数据日期满足预期

风险：

- 可能用旧数据成功跑完当天任务
- 后续即使新数据到了，也会因为幂等逻辑被跳过

要求：

- 数据日期必须参与运行判定
- 数据不新鲜时，系统应 fail closed，而不是继续成功运行

### P1. 对外业绩表述与当前可执行结果不一致

问题：

- 公开文档中的关键回测成绩与当前可执行脚本结果不一致

风险：

- 伤害系统可信度
- 让治理讨论建立在过期数据之上

要求：

- 文档指标必须由回测脚本自动导出
- 不允许手工维护核心业绩数字

### P2. 认证与启动校验仍偏原型化

问题：

- JWT secret 存在弱默认值
- 密码哈希仍是 SHA-256
- 启动时未强制执行配置校验

风险：

- 金融控制面长期暴露风险过高
- 错误配置可带病启动

要求：

- 强制设置密钥
- 使用 bcrypt 或 argon2
- 启动即执行配置校验，失败则拒绝运行

## 五、文档级硬约束

从本文件生效起，息壤的对外和对内描述必须遵守以下口径：

### 入金

- 入金是唯一允许自动化的主资金流动作
- 入金先进入 Stability
- 只有在不破坏风控边界时，系统才把资金从 Stability 转入 Core

### 出金

- 出金不是自动化动作
- 系统只提供申请、预览、审批、留痕
- 真正卖出资产、划转资金、离开账户，必须人工完成

### Alpha / Lab

- Alpha 不是主仓增强版
- Alpha 是独立实验仓
- Alpha 的收益、回撤、状态、交易记录必须与主仓分开核算

## 六、实施优先级

### 第一优先级：先堵住错误

1. 关闭公开初始化管理员入口
2. 禁止任何自动出金批准/执行
3. 把数据新鲜度改成硬校验
4. 启动时执行配置校验

### 第二优先级：再硬化边界

1. 将 Alpha 改为独立表和独立账户作用域
2. 禁止 Alpha 共享主仓默认 `portfolio_id`
3. 将主仓与实验仓的 API、审计、净值完全拆开

### 第三优先级：最后统一口径

1. 用脚本自动生成回测结论
2. 更新 README 和系统设计文档
3. 把"全自动"表述改为"核心调仓自动，出金永远人工"

## 七、一句话总纲

> 息壤不是用来追求更刺激收益的。
> 它是家族资产的低错误率控制系统。
> 入金可以自动化，出金必须人工化，实验必须隔离，所有收益都必须让位于长期稳健。
