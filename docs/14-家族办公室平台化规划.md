# 家族办公室平台化规划

> 核心判断：息壤不应演化成“每个成员一套投资机器人”。
> 正确方向是“统一资金池 + 份额记账 + 多用户权限 + 强审计”。

## 当前状态

息壤当前已经具备家族办公室内核：

- `PORTFOLIOS` 已支持多市场组合，例如 `us`、`cn`
- API 已有登录用户与角色：`admin / member / viewer`
- 治理层已有入金、出金申请、审批、执行、审计日志
- Core / Stability / Alpha 三层资产结构已经成型
- 券商同步、对账、执行审计已进入最小真实闭环

但当前系统仍然更像“一个人的资产操作系统”。主要原因是：

- `portfolio_id` 当前表示市场组合，不表示家族成员账户
- 入金记录只保存 `depositor` 字符串，没有真正的成员账户归属
- 用户角色只控制能否访问接口，没有控制“能看到谁的钱”
- 每个成员的资产权益、份额、收益归属还没有独立账本

## 产品原则

### 1. 稳定优先

家族办公室不是高频交易系统，也不是社交投资平台。它的目标是：

- 较少操作
- 极少影响
- 极少注意力
- 账本可信
- 风控先于收益

任何新增能力，只要会显著增加日常维护负担，都应默认拒绝。

### 2. 平台和用户解耦

平台负责：

- 市场数据
- 投资池净值
- 资产配置
- 风控状态
- 再平衡建议与执行
- 券商同步与对账
- 审计日志

用户拥有：

- 登录身份
- 家族成员身份
- 一个或多个资产账户
- 对投资池的份额
- 自己的流水、报表、申请与审批权限

用户不直接拥有策略引擎。

### 3. 统一资金池，按份额记账

所有成员资金进入统一投资池。成员不是各自运行一套 `portfolio`，而是持有投资池份额。

这避免了：

- 每个用户单独调仓
- 每个用户单独对账
- 每个用户单独处理券商订单
- 每个用户单独计算风控
- 成员数量增长后维护成本线性爆炸

对家族办公室来说，真正可维护的模型是：

```text
用户账户 -> 持有份额 -> 投资池 -> 统一配置 / 统一风控 / 统一执行
```

### 4. 收益通过份额净值自然归属

平台不应每天把“收益”拆成一笔笔用户流水。正确模型是基金会计：

```text
投资池总净值 NAV = 现金 + 持仓市值 - 费用/负债
每份净值 share_price = NAV / 总份额 shares_outstanding
用户资产 = 用户持有份额 * 当前每份净值
```

入金不是“加入本金池后按金额分收益”，而是“按当前份额净值申购份额”。出金不是“提走本金和收益”，而是“赎回一部分份额”。

这样可以保证：

- 老成员不会被新入金稀释
- 新成员不会白拿历史收益
- 后续涨跌自动按份额承担
- 用户资产展示可以从份额和净值复算
- 日常不需要人工分配收益

例子：

```text
初始池子 NAV = 1,000,000
总份额 = 10,000
每份净值 = 100

张三入金 100,000
获得份额 = 100,000 / 100 = 1,000 份

之后池子涨到 NAV = 1,210,000
总份额 = 11,000
每份净值 = 110

张三资产 = 1,000 * 110 = 110,000
张三收益 = 10,000
```

MVP 估值规则建议：

- 入金申请不锁定份额价格
- 管理员确认真实到账后，使用最近一次已锁定的 `pool_nav_snapshot`
- 份额生效后写入 `ledger_entries`
- 非交易日不创造临时价格，沿用最近一次锁定净值

后续可以升级为：

```text
T 日到账确认
使用 T 日收盘后的池子净值
T+1 生效份额
```

原则：用户收益不靠人工分配，靠份额净值变化自然归属。

## 核心业务模型

### 家族办公室

`family_offices`

代表一个家族办公室实体。短期可以只有一个默认家族，长期支持多租户。

关键字段：

- `id`
- `name`
- `base_currency`
- `status`
- `created_at`

### 登录用户

`api_users`

继续作为登录身份表，但需要从“角色表”升级为“身份入口”。

应补充：

- 是否绑定家族成员
- 是否启用 MFA
- 最近登录时间
- 账户锁定状态

### 家族成员

`family_members`

代表真实经济权益人或受益人。登录用户和家族成员不能混为一谈。

例子：

- 父亲
- 母亲
- 子女
- 家族信托
- 控股实体

关键字段：

- `id`
- `family_office_id`
- `display_name`
- `member_type`
- `risk_profile`
- `status`

### 资产账户

`capital_accounts`

一个成员可以有多个资产账户。

例子：

- 张三个人账户
- 张三教育金账户
- 家族信托账户
- 共同账户

关键字段：

- `id`
- `family_office_id`
- `member_id`
- `account_name`
- `base_currency`
- `status`

### 用户到账户授权

`account_permissions`

控制某个登录用户能看或操作哪些账户。

建议权限：

- `view`
- `deposit_request`
- `withdraw_request`
- `approve_withdrawal`
- `manage_account`
- `admin`

### 投资池

`investment_pools`

投资池是平台运行的资金池，继承现在 `portfolio_id` 的投资含义。

例子：

- `core_us`
- `core_cn`
- `stability_usd`
- `stability_cny`
- `alpha_us`

关键字段：

- `id`
- `family_office_id`
- `portfolio_id`
- `pool_type`
- `currency`
- `nav`
- `shares_outstanding`
- `share_price`
- `status`

### 账户持有份额

`account_pool_positions`

记录每个资产账户持有每个投资池多少份额。

关键字段：

- `account_id`
- `pool_id`
- `shares`
- `cost_basis`
- `updated_at`

### 总账流水

`ledger_entries`

这是平台化后最重要的表。所有资金变化都必须进总账。

流水类型：

- `DEPOSIT_REQUESTED`
- `DEPOSIT_CONFIRMED`
- `POOL_SUBSCRIPTION`
- `POOL_REDEMPTION`
- `WITHDRAWAL_REQUESTED`
- `WITHDRAWAL_APPROVED`
- `WITHDRAWAL_EXECUTED`
- `FEE_CHARGED`
- `NAV_REVALUED`
- `MANUAL_ADJUSTMENT`

关键字段：

- `id`
- `family_office_id`
- `account_id`
- `pool_id`
- `entry_type`
- `amount`
- `currency`
- `shares_delta`
- `share_price`
- `actor`
- `source_ref_type`
- `source_ref_id`
- `created_at`

原则：任何用户资产展示都应能从总账和份额表解释出来。

## 权限模型

现有 `admin / member / viewer` 不够细。建议升级为平台角色 + 账户权限两层。

### 平台角色

- `platform_admin`：系统管理员，管理用户、角色、全局配置
- `family_principal`：家族负责人，可看全局资产，可参与重大审批
- `investment_manager`：投资管理员，可处理入金确认、调仓、券商同步
- `operations`：运营角色，可处理流水、报表、基础审核
- `auditor`：审计只读
- `member`：普通成员，只能看被授权账户

### 账户权限

平台角色决定系统级能力，账户权限决定可见资金范围。

普通成员不能通过传入别人的 `account_id` 或 `portfolio_id` 越权查看资产。

所有资产类接口必须先做授权检查。

## 用户工作流

### 成员登录

成员登录后看到：

- 自己有权限的账户列表
- 总资产
- 投资池配置
- 收益曲线
- 近期流水
- 出金申请状态

成员不看到：

- 别人的资产
- 券商真实账号明细
- 系统级风控配置
- 未授权的全局报表

### 入金

第一阶段建议仍然人工确认到账，不自动接银行。

流程：

1. 成员提交入金申请
2. 系统生成待确认记录
3. 管理员确认真实到账
4. 系统按当前投资池净值申购份额
5. 写入总账
6. 更新账户持有份额
7. 成员看到资金已入账

### 出金

出金是治理动作，不是自动交易动作。

流程：

1. 成员发起出金申请
2. 系统检查可用份额和冷却规则
3. 按规则进入审批
4. 审批通过后，由管理员执行
5. 系统赎回份额或从 Stability 池扣减
6. 写入总账与审计日志

永久禁止默认自动化：

- 出金批准
- 出金执行
- 跨账户划款
- 跨券商迁仓

## 和现有系统的映射

### `portfolio_id`

保留为投资池或市场策略维度，不再直接代表用户账户。

短期映射：

- 当前 `us` -> `investment_pools.core_us`
- 当前 `cn` -> `investment_pools.core_cn`

### `portfolios`

继续作为策略运行状态表，但未来应改名或语义收窄为 `strategy_portfolios`。

它保存的是投资池层面的：

- Core 持仓
- Stability 资金池
- 风控状态
- high water mark
- 再平衡状态

不是某个成员个人资产。

### `deposit_records`

短期兼容，长期迁移到：

- `deposit_requests`
- `ledger_entries`
- `account_pool_positions`

### `withdrawal_requests`

保留治理流程，但增加：

- `account_id`
- `member_id`
- `requested_by_user_id`
- `approved_by_user_id`
- `source_pool_id`

## 分阶段落地计划

### Phase 1：账户与权限解耦

目标：让“谁能看谁的钱”成为数据库事实。

交付：

- 新增 `family_members`
- 新增 `capital_accounts`
- 新增 `account_permissions`
- API 增加“我的账户”接口
- 现有 `api_users` 绑定成员
- 所有 portfolio/governance 查询增加账户权限检查

验收：

- 普通成员只能看到自己账户
- 管理员能看到全局
- 测试覆盖越权访问

### Phase 2：入金账本化

目标：入金从“字符串记录”升级为“账户流水 + 份额申购”。

状态：已落地最小闭环。

交付：

- 新增 `ledger_entries`
- 新增 `deposit_requests`
- 新增 `pool_nav_snapshots` 的最小锁定净值口径
- 新增 `account_pool_positions` 的账户份额持仓
- 成员可提交入金申请，申请本身不移动组合资金
- 管理员确认真实到账后，使用最近一次锁定份额净值申购份额
- 入金确认后写总账
- 入金确认后更新账户份额
- 保留旧接口兼容，但内部改走账户模型

验收：

- 每笔入金都能追溯到申请人、确认人、账户、投资池、份额价格
- 用户资产能从份额表计算出来
- 旧 `deposit_records` 继续保留，但会补充 `account_id`、`deposit_request_id`、`ledger_entry_id`、`shares_issued`、`share_price`

当前实现口径：

```text
DEPOSIT_REQUESTED:
  只写申请、总账事件与审计，不改变 portfolio NAV

DEPOSIT_CONFIRMED:
  管理员确认到账
  使用最近锁定 share_price 计算 shares_issued
  更新 account_pool_positions
  更新 pool_nav_snapshots
  写 DEPOSIT_CONFIRMED 与 POOL_SUBSCRIPTION 两类总账流水
```

### Phase 3：投资池份额制

目标：完成平台和用户资产的真正解耦。

交付：

- 新增 `investment_pools`
- 扩展 `pool_nav_snapshots` 为每日正式净值任务
- 扩展 `account_pool_positions` 到出金、费用、调整等全场景
- 每日任务更新投资池净值
- 用户资产按 `shares * share_price` 计算
- 管理端展示全局 AUM

验收：

- `us/cn` 仍然只运行一次策略
- 10 个用户不会产生 10 套调仓和对账任务
- 每个用户看到自己的份额和收益

### Phase 4：出金与报表平台化

目标：让成员端真正可用。

交付：

- 出金申请绑定账户和份额
- 出金审批矩阵
- 月度成员报表
- 家族全局报表
- 审计导出

验收：

- 成员可自助查看资产与流水
- 家族负责人可看全局但不需要天天操作
- 出金全链路可审计

## 非目标

短中期不做：

- 每个成员独立策略
- 每个成员独立券商账户自动交易
- 自动银行转账
- 自动出金执行
- 成员之间自由转让份额
- 社交化投资功能
- 高频交易功能

这些都会显著提高系统复杂度，不符合息壤“稳定、少操作、低注意力”的本质。

## 关键风险

### 1. 账本不可逆复杂化

一旦开始记录用户权益，账本设计必须保守。宁可多一张审计表，也不要允许静默改余额。

### 2. 权限越权

平台化后最大安全风险不是交易亏损，而是用户看到不该看到的资产。

所有资产相关接口必须以 `account_id` 做授权边界。

### 3. 份额价格口径

必须明确：

- 入金用哪个时间点的份额价格
- 出金用哪个时间点的份额价格
- 非交易日如何处理
- 手续费如何归属

这些规则必须在实现前写入测试。

## 最小可行版本

最小版本不需要漂亮前端，也不需要银行自动化。

只需要做到：

1. 管理员创建成员和账户
2. 成员登录后只看自己的账户
3. 成员发起入金申请
4. 管理员确认到账
5. 系统按统一资金池份额记账
6. 成员看到自己的资产和流水
7. 管理员看到全局 AUM

做到这一步，息壤就从个人系统跨过了平台化门槛。

## 推荐下一步

先做 Phase 1。

理由：不先把账户、成员、权限边界定死，后面入金、出金、份额、报表都会继续把 `username`、`portfolio_id`、`depositor` 混在一起，越改越乱。

第一张工程任务单应是：

> 建立家族成员、资产账户、账户授权三张表，并让普通用户只能读取自己授权账户下的资产视图。
