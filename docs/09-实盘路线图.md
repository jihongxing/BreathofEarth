# 从仿真到实盘：四步路线图

> 本路线图只讨论调仓执行自动化。
> 出金不在自动化范围内，长期原则保持不变：只能发起请求，后续人工审批、人工执行。

核心引擎的逻辑完全不变，只是"执行层"在不同阶段切换不同的实现。

## Phase 1: Paper Trading（历史阶段）

执行器：`PaperExecutor`
配置：`XIRANG_EXECUTOR=paper`（默认）

行为：
- 所有交易都是虚拟的，数据库里改数字
- 用 Yahoo Finance 的收盘价作为"成交价"
- 扣除 0.1% 的模拟摩擦成本

验证目标：
- 状态机逻辑是否正确
- 风控触发是否合理
- 通知推送是否正常
- 报告数据是否准确

周期：3-6 个月

退出条件：
- 运行成功率 > 99%
- 风控触发合理（无误触发）
- 报告指标与回测一致

## Phase 2: 人工执行期（历史阶段）

执行器：`ManualExecutor`
配置：`XIRANG_EXECUTOR=manual`

行为：
- 系统生成具体指令："卖出 12 股 SPY，买入 8 股 TLT"
- 通过 Telegram / 微信推送给你
- 你在券商 APP 上手动执行
- 执行完成后在系统中确认，录入实际成交价

新增能力：
- 具体股数计算（目标金额 / 当前价格，取整）
- 先卖后买排序（卖出回笼资金后再买入）
- 指令 vs 实际的滑点统计

验证目标：
- 指令是否合理（股数、方向、金额）
- 真实滑点与模拟滑点的差距
- 手动执行的操作负担是否可接受（一年约 50 次）

周期：6-12 个月

退出条件：
- 真实滑点 < 0.2%
- 无一次指令错误
- 操作流程顺畅

## Phase 3: 最小真实执行期

执行器：`BrokerExecutor(auto_confirm=False)` 或 `BrokerExecutor(auto_confirm=True)`
配置：`XIRANG_EXECUTOR=semi_auto/auto`，主券商 `IBKR`，备券商 `Futu`

行为：
- 系统已能通过券商 API 提交最小标准再平衡订单
- 执行前必须通过券商同步硬闸门与白名单矩阵
- 执行后必须通过对账闭环与回执检查
- 该自动化仅适用于 Core 调仓与再平衡，不适用于出金

新增能力：
- 券商 API 连接（IBKR / 富途）
- 订单提交、状态跟踪、撤单、回执解析
- 账户持仓同步（每天从券商拉取真实数据）
- 执行审计流水持久化

验证目标：
- API 连接稳定性
- 订单执行可靠性
- 持仓同步准确性
- 执行后对账与审计链完整性

当前建议：

- 不急着推进下一阶段
- 保持最小真实交易能力即可
- 把新增时间优先投到文档、观察、验证和边界固化

## Phase 4: 核心调仓全自动（暂不推进）

执行器：`BrokerExecutor(auto_confirm=True)`
配置：`XIRANG_EXECUTOR=auto`，`XIRANG_BROKER=ibkr`

行为：
- Core 调仓自动执行，无需人工干预
- 只在风控触发时推送通知
- 每天自动同步券商持仓
- 出金仍然停留在治理层，由人工处理

安全机制：
- 单笔订单金额上限（防止 bug 导致全仓操作）
- 日交易次数上限（防止死循环）
- 异常价格检测（防止闪崩误操作）
- 所有订单记录可审计

## 切换方式

只需要改一个环境变量：

```bash
# Phase 1 → Phase 2
echo "XIRANG_EXECUTOR=manual" >> .env

# Phase 2 → Phase 3
echo "XIRANG_EXECUTOR=semi_auto" >> .env
echo "XIRANG_BROKER=ibkr" >> .env

# Phase 3 → Phase 4
# 把 semi_auto 改成 auto
sed -i 's/semi_auto/auto/' .env
```

注意：

- 真实下单还需要显式开启券商级开关，例如 `IBKR_ENABLE_ORDER_SUBMISSION=1`
- Core 真实执行还必须额外开启 `XIRANG_ENABLE_LIVE_CORE_EXECUTION=1`
- 每次真实 Core 验证必须写入 `XIRANG_LIVE_CORE_APPROVAL_ID`，作为人工批准引用
- Futu 同样需要显式交易开关与交易解锁密码
- 默认推荐保持关闭，除非是在小额验证窗口内

## Stage 9.5：影子同步与结构性审计

这是当前阶段。

目标：

- 不下真实订单
- 连续观察 90/10 生产候选目标账本
- 记录真实或本地报价下的滑点估计
- 只读抓取 IBKR 保证金快照
- 证明券商真实保证金和利差结构没有击穿回测假设

命令：

```bash
python -m live.stage95_shadow_runner --aum 2000000 --no-broker --skip-db
python -m live.stage95_observation_summary --shadow-dir data/shadow --expected-cycles 60
```

接通券商后，可以去掉 `--no-broker`，让 `shadow_sync` 使用只读券商报价和账户快照。该运行器会同时写出 `latest_shadow_sync.json`、`latest_margin_snapshot.json` 与 `latest_stage95_cycle.json`。汇总器会写出 `latest_stage95_observation_summary.json`，统计 60 日覆盖率、券商不可用次数、滑点、保证金字段覆盖率和连续异常 streak。如果券商不可达、保证金字段缺失，或报告超过过期阈值，系统必须返回 `WARNING / UNAVAILABLE / ATTENTION`，不能显示为安全。

这些报告已经通过 FastAPI 只读接口暴露到前端观察面板。前端只负责展示后端审计结论，不允许提供交易、加杠杆或 Shadow 转 Live 入口。Stage 9.5 完成前，任何 Portfolio Margin 讨论都只能停留在结构性观察和人工审计层面。

更完整的方案与后续开发准则见：[生产候选方案与后续开发指南](D:/codeSpace/BreathofEarth/docs/15-%E7%94%9F%E4%BA%A7%E5%80%99%E9%80%89%E6%96%B9%E6%A1%88%E4%B8%8E%E5%90%8E%E7%BB%AD%E5%BC%80%E5%8F%91%E6%8C%87%E5%8D%97.md)。

## 永久人工边界

以下动作不进入 Phase 4：

- 出金审批
- 出金执行
- 资金离开券商/银行账户
- Lab 向 Core 的资金回流批准

这些动作始终属于人工治理，而不是自动执行。
