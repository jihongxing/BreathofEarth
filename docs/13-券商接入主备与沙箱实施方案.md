> 目标不是“更频繁交易”，而是把 Core 的真实账本、真实成交、真实回执闭环补齐。
> 系统默认追求少打扰、少犯错、少切换；收益提升模块永远排在账本真实性之后。

## 一、边界先定死

### 1. 永久不自动化的动作

- 出金批准
- 出金执行
- 账户间划款
- 跨券商迁仓
- Alpha 盈利回流 Core 的批准

这类动作都属于治理，不属于自动交易。
系统只允许发起申请、记录审批链、记录人工处理结果，不允许直接代替人完成资金离场。

### 2. 可以逐步自动化的动作

- Core 日常行情拉取
- Core 风控检测
- Core 再平衡指令生成
- 券商只读同步
- 券商影子下单比对
- 小额真实调仓执行

### 3. 主备的真实含义

主备不是“主券商挂了立刻自动切备券商继续下单”。

对家族资产系统，主备必须满足以下前提后，才允许切换：

- 持仓对齐
- 现金对齐
- 成本对齐
- 税务口径对齐
- 历史成交与回执可追溯
- 切换经过人工治理批准

因此，本系统的主备首先是“账本主备 + 对账主备 + 能力主备”，最后才是“执行主备”。

## 二、市场与券商策略

### 1. 市场优先级

- 美国市场是主战场，优先完成真实执行闭环
- 中国市场是大本营，必须同步建设，但不为了赶进度牺牲边界清晰度

### 2. 建议券商组合

第一阶段落地建议：

- 美国主券商：IBKR
- 备用券商：Futu
- 沙箱适配：PaperAdapter

中长期预留：

- 中国本土实盘主腿：国内券商 + QMT / PTrade 适配器

原因很简单：

- `IBKR` 更适合做美国主链路
- `Futu` 适合作为中美覆盖过渡层与备链路
- 真正的中国本土实盘能力，长期不应只押在“港美平台顺带支持”的能力上

## 三、必须拆开的三套环境

### 1. Live

- 真资金
- 真订单
- 真回执
- 真对账

### 2. Shadow

- 读真实账户
- 生成拟执行订单
- 不下单
- 每日比较“系统应该怎么做”与“账户真实发生了什么”

### 3. Sandbox

- 模拟账户或本地纸面账户
- 用于开发、联调、回归测试、故障演练
- 不进入正式收益汇报

原则：

- Shadow 与 Sandbox 不能写入正式收益报表
- Live 与 Sandbox 不能共用账本
- 任何“模拟利润”不能进入正式排行榜或对外收益汇报

## 四、四步实施节奏

### Phase 1：只读同步 + 对账先行

目标：

- 接通券商账户信息
- 拉取现金、持仓、订单、成交、回执
- 建立本地对账报告

产出：

- `BrokerAdapter`
- `Reconciliation`
- 每日账户快照
- 差异告警

上线标准：

- 不下任何真实单
- 连续运行稳定
- 对账差异可解释

### Phase 2：Shadow Run

目标：

- 系统每天产出拟执行单
- 与真实账户状态比较
- 与人工执行结果比较

产出：

- `ShadowRun`
- 拟执行订单日志
- 拟执行与实执行偏差报告

上线标准：

- 连续一段时间无明显偏差
- 数据口径统一
- 对账闭环稳定

### Phase 3：小额真实资金执行

目标：

- 在严格风控下，小额执行 Core 调仓
- 仅限白名单市场、白名单资产、白名单动作

约束：

- 单笔金额上限
- 单日成交次数上限
- 单日换手上限
- 只允许标准再平衡订单
- 数据过期、成交偏差过大、券商回执缺失时自动升级人工介入

### Phase 4：双市场生产化

目标：

- 美国主链路生产化
- 中国链路同步生产化
- 备链路持续只读与能力校验

注意：

- 主备切换必须经过人工治理
- 不允许把“自动切备”做成默认行为

## 五、执行层目标架构

```text
PortfolioEngine / RiskEngine
            |
            v
      BaseExecutor
            |
    +-------+--------+
    |                |
    v                v
PaperExecutor   BrokerExecutor
                     |
      +--------------+--------------+
      |              |              |
      v              v              v
BrokerAdapter   Reconciliation   ShadowRun
      |
  +---+---+--------------------+
  |       |                    |
  v       v                    v
IBKR   Futu / QMT         PaperAdapter
```

职责划分：

- `BrokerAdapter`
  - 只负责和券商交互
  - 不负责调仓决策
- `PaperAdapter`
  - 提供统一的模拟券商接口
  - 用于沙箱和影子测试
- `Reconciliation`
  - 比较本地账本与券商真实状态
  - 发现差异即 fail-closed
- `ShadowRun`
  - 只产生命令，不提交交易
  - 为 Phase 2 服务

## 六、推荐实施顺序

### Step 1：基础协议

- 定义账户快照、持仓快照、订单请求、订单回执模型
- 定义 broker mode：`read_only / paper / live`
- 定义 broker role：`primary / backup / sandbox`

### Step 2：适配器骨架

- `IBKRAdapter`
- `FutuAdapter`
- `PaperAdapter`

此时只要求接口通，不要求实盘可下单。

### Step 3：对账能力

- 账户总资产对账
- 现金对账
- 逐资产市值对账
- 缺失持仓与多余持仓检测

### Step 4：影子运行

- 读取真实账户
- 生成拟执行单
- 输出偏差报告

### Step 5：再接真实执行

- 先美国、后中国
- 先小额、后正式
- 先主链路、后备链路

## 七、代码落地要求

- 正式收益汇报只认真实成交 + 券商回执 + 现金流水闭环
- Shadow/Sandbox 结果不得混入正式汇报
- 任何新券商接入，必须先实现只读与对账，再谈下单
- 新适配器不能直接侵入 Core 决策层
- API 与账本边界优先，收益增强模块延后

## 八、这一轮开发范围

本轮原始计划只做以下事情：

- 新增券商接入设计文档
- 新增 `BrokerAdapter / PaperAdapter / Reconciliation / ShadowRun` 骨架
- 将工厂切到新的基础设施入口
- 补最小测试，确保主备/沙箱选择与对账逻辑可回归

原始计划明确不做：

- 真实券商下单
- 自动主备切换
- 跨券商迁仓
- 正式税务/收益优化增强

## 九、状态修正

最初版本写明“本轮明确不做真实券商下单”，这是当时的开发边界。

但后续实际实现已经推进到更靠前的一步：

- 已接入 `IBKR / Futu` 的最小真实交易协议
- 已接入执行前硬闸门与执行后对账
- 已接入真实执行审计留痕

这不代表系统进入“积极扩张的实盘阶段”，恰恰相反，当前推荐状态是：

- 保持默认关闭真实下单开关
- 仅在白名单市场、白名单资产、白名单动作下做小额验证
- 以观察期和审计期代替功能继续扩张
